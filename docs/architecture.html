What is AgentLens?

Moonshot ideas, evaluated end-to-end

Submit any moonshot product concept. A LangGraph pipeline of specialist AI agents analyzes it in parallel and produces a Kill Shot experiment — the cheapest test of the most critical assumption.

🎯
Designed for X's Evaluation Framework
Models the Moonshot Factory's 3-criteria gate (real problem, feasible solution, available technology) that every X project must clear before resources are committed.
⚡
Parallel Agent Execution
Five specialist agents (technical, market, risk, cost, RAG) run simultaneously via LangGraph's Send API fan-out — not sequentially. Reduces latency from 5× to 1× the slowest agent.
👤
Human-in-the-Loop
After all 5 agents complete, execution pauses for expert review. Domain experts can approve, reject, or add correction notes before the Kill Shot designer runs.
🔬
Kill Shot Experiment
The final agent synthesizes all outputs and designs the single cheapest experiment to test the most critical assumption — producing an actionable output, not just a report.
📊
Evaluation Layer
LLM-as-judge (per-criterion) + RAGAS metrics + benchmark sweeps. Detects quality regressions, measures RAG faithfulness, and compares model configurations programmatically.
🛡️
4-Layer Guardrails
Defense in depth: prompt instructions, Pydantic output schemas, pre-generation intent classifier, and post-generation rule checks. Single-layer guardrails are easily bypassed.

🛡️
Intent Classifier
Pre-generation safety + input type classification
Idle

What it does

  • Classifies input as safe / borderline / harmful
  • Determines input_type: "idea" or "chat"
  • Returns both in one LLM call using Pydantic InputAnalysis schema

Routing

  • harmful → rejection_report (terminates)
  • chat → chat_responder (friendly reply)
  • idea + safe → moonshot_evaluator

Why combine safety + type in one call?

Two separate LLM calls would double latency on the hot path. A single InputAnalysis Pydantic schema forces the model to reason about both in one forward pass. The combined call adds ~200ms — far less than two sequential calls.

🌙
Moonshot Evaluator Gate
3 pass/fail criteria — fails fast before expensive parallel agents run
Idle

Three Criteria

  • Is the problem real and urgent?
  • Is the solution 10× better, not incremental?
  • Is the enabling technology available? (TRL check)

Tools Used

  • DuckDuckGoSearchRun — market evidence
  • ArxivQueryRun — research papers
  • WikipediaQueryRun — context
  • ChromaDBRetriever — domain KB

Why gate first?

5 parallel LLM calls + kill shot = ~$0.10 per run. A fast rejection at the gate costs ~$0.005 and saves the full pipeline cost on ideas that obviously don't qualify. The gate also produces a structured rejection reason, not just a binary fail signal.

Why here?

This is the highest-value intervention point: all 5 agent outputs are available for review, but the Kill Shot hasn't run yet. Errors in parallel outputs (wrong TRL, incorrect cost) propagate directly into the experiment design — a domain expert can correct them before they cascade.

What the human can do

  • Approve + comment — correction text injected into Kill Shot prompt

LangGraph Implementation

The human_review node is empty by design — it exists solely as the interrupt_after target. After interruption, the frontend calls POST /evaluate/{id}/resume with the decision, the state is updated via graph.update_state(), and the graph resumes from checkpoint.

Kill Shot Experiment Designer
The synthesis agent — reads ALL prior outputs to find the weakest assumption
Idle

Outputs (KillShotExperiment)

  • why_this_assumption — cross-agent justification
  • experiment_description — cheapest test design

Why runs last

Needs the full picture from all specialists to identify the weakest link across technical, market, risk, and cost dimensions. Running in parallel would force speculation without full context — producing a guess, not a synthesis.

What makes this different from a summary agent

A summary agent produces a report. The Kill Shot designer produces an action — a specific, costed experiment with measurable success and failure criteria. This forces the system to commit to a recommendation, not hedge with a comprehensive list of concerns.

Memory Architecture

Three types of memory, three failure modes

Production cognitive architectures implement all three memory types. Each maps to a distinct component with distinct failure modes — missing any one degrades the system in a different way.

Episodic Memory
What happened in this run
Memory of specific past events — the full execution history of this evaluation and prior evaluations by thread_id.
Implementation: LangGraph checkpointer (SqliteSaver locally, RedisSaver / PostgresSaver in production). Full GraphState persisted after every node boundary.
LangGraph Checkpointer
Failure Mode

Checkpoint loss → run unrecoverable; must restart from raw idea input. HITL state lost — human cannot resume hours after reviewing.

Semantic Memory
Long-term factual knowledge
What the system knows about the world — TRL benchmarks, CAPEX/OPEX databases, market reports, past kill shot experiment designs.
Implementation: ChromaDB vector store with two source types: "domain" (ingested at setup) and "portfolio" (past project docs). Retrieved by the RAG agent via cosine similarity (top-k=5, threshold=0.75).
ChromaDB Vector Store
Failure Mode

Retrieval miss → agents hallucinate domain facts they should have grounded. Embedding model drift after dependency upgrade → all similarity scores meaningless.

Procedural Memory
How to do things
Skills, evaluation rubrics, the moonshot framework — the system's implicit knowledge of how to reason, what to output, and what to avoid.
Implementation: System prompts + Pydantic tool schemas + few-shot examples baked into each agent. The 3-question gate is procedural memory made explicit in code. LangSmith stores every prompt template version.
Prompts + Schemas
Failure Mode

Prompt drift after a model update → agents silently stop following the rubric. No error thrown — the LLM-as-judge must detect this via score regression.

Safety & Reliability

4-Layer Guardrails

Single-layer guardrails are easily bypassed. Defense in depth means each layer catches what the previous one misses.

Hover a layer to explore its mechanism.

1
Prompt-level
Explicit "must" and "must not" rules in every agent's system prompt. Cheapest layer — zero latency cost. Fails when model ignores instructions (jailbreak, unusual input).
2
Schema-level (Pydantic)
Required Pydantic fields force explicit uncertainty bounds, evidence citations, and assumption lists. Model cannot produce output that passes validation without filling these — overconfidence literally fails validation.
3
Pre-generation (Intent Classifier)
Runs before any specialist agent. Blocks harmful inputs (weapons, mass surveillance, dual-use) at the graph entry point — no expensive LLM calls consumed on bad inputs.
4
Post-generation (Rule Checks)
After agents complete: verbatim retrieval detection (>40 words, >80% token overlap with source doc), null field detection for required fields, inter-agent consistency check on shared assumptions (TRL level, market size order).

What Agents Must Never Produce

⛔Speculative financial projections without uncertainty bounds and stated assumptions
⛔Feasibility claims without TRL citation and retrieved evidence
⛔Verbatim reproduction of retrieved documents (copyright / legal risk)
⛔Hallucinated citations — only documents actually retrieved may be cited
⛔Definitive regulatory or safety certification claims
⛔Uncritical validation — risk agent must always produce ≥3 distinct risks
⛔Internal system details (prompt text, model names, config) in output

Architecture Rationale

Key Design Decisions

Every choice has a reason. Click any card to see the alternative considered and why it was rejected.

Why LangGraph over sequential pipeline?
StateGraph with parallel fan-out and checkpointing
+
5 independent agents serialized in a chain multiply latency by 5×. LangGraph's Send API fans them out simultaneously, cutting latency to 1× the slowest agent. Checkpointing means a crash at node 4 resumes from the checkpoint — not from zero. Conditional edges make the moonshot gate a first-class citizen, not an if/else in application code.
Alternative Rejected

Sequential LangChain pipeline, custom async Python, CrewAI, AutoGen. None provide stateful checkpointing + native HITL + conditional routing in a single graph primitive.

Why Pydantic for all agent outputs?
Typed schemas + tool-use mode eliminate a whole class of errors
+
Free-form LLM text fails unpredictably in pipelines — missing fields cause KeyError at runtime, not validation time. Pydantic v2 + Claude's tool-use mode guarantees valid, typed output. Required fields like uncertainty_level and evidence_citations also force the model to be explicit about confidence, directly countering overconfident hallucination.
Alternative Rejected

Regex/JSON parsing of LLM text. Fragile — any model variation in formatting causes a parse failure. No forced schema adherence.

Why chromaDB as vector store?
Zero infra locally, abstracted for easy swap
+
ChromaDB runs as a Python library — no database server, no configuration, no infrastructure cost. All retrieval logic is behind vector_store.py. In production, swapping to Pinecone, Weaviate, or Vertex AI Matching Engine is one file change — no agent code changes required.
Alternative Rejected

Pinecone (requires account + managed infra for prototyping), FAISS (no built-in metadata filtering), Weaviate (requires Docker).

Why a dedicated Kill Shot agent, not a summary?
Actionable output over comprehensive reporting
+
A summary agent produces a report. The Kill Shot designer produces an action — a specific experiment with cost, duration, and measurable criteria. This forces commitment to a recommendation. The agent synthesizes across technical, market, risk, and cost outputs to find the weakest link — it must run sequentially after all 5 parallel agents, not in parallel.
Alternative Rejected

Aggregator/synthesis agent that summarizes all outputs. Produces comprehensive coverage but no actionable recommendation — not useful for deciding whether to fund further exploration.

Why SSE over WebSocket for streaming?
Unidirectional flow matches the streaming pattern exactly
+
SSE is unidirectional server→client — which is exactly what pipeline event streaming requires. It runs over standard HTTP (no upgrade handshake, works through proxies). EventSource auto-reconnects with Last-Event-ID. The one bidirectional interaction (HITL resume) is a plain HTTP POST — no persistent socket needed.
Alternative Rejected

WebSocket adds bidirectionality we don't need on the stream channel, plus a separate upgrade connection. Polling adds latency and server load.

Why LLM-as-judge per-criterion, not holistic?
Eliminates positional and self-serving bias
+
Holistic scores are positionally biased (judge favors the first answer presented) and self-serving (a model favors outputs from its own family). Per-criterion scoring with randomized answer order breaks both biases. Each criterion can detect a specific failure mode — overconfidence penalty catches a different problem than coherence score.
Alternative Rejected

Single holistic 1–10 score. Cannot distinguish between an idea that's technically excellent but poorly costed vs. one that's coherent but overconfident.

Why not use MCP for tools?
LangGraph tool interface is tighter and better integrated
+
MCP adds a client-server boundary with network latency and a new failure mode inside the agent loop. The tools we need (Arxiv, Wikipedia, ChromaDB) are already LangChain tools with no server setup. MCP is the right choice when tools are shared across teams or model frameworks — not for a single-team production pipeline.
When MCP is right

Multi-team production deployments where tools are owned and served by different services. When the same tool needs to be shared across multiple agent frameworks. As a future extension point — our tool abstractions make the migration trivial.

Why custom agents, not pre-built templates?
Schema control + observability + evaluability
+
Pre-built agents (ReActAgent, OpenAIFunctionsAgent) are opaque — we cannot inspect their reasoning strategy or guarantee their output format. Custom agents give us: auditable system prompts, Pydantic output contracts, and every prompt version stored in LangSmith. We reuse infrastructure (LangGraph primitives, tool wrappers) but build all reasoning.
What we do reuse

LangGraph StateGraph primitives, LangChain tool wrappers for Arxiv/Wikipedia/DuckDuckGo, RAGAS evaluation library. Rule: reuse infrastructure; build reasoning.

Why HITL pause after parallel agents, not before?
Maximum context for human review, minimum wasted compute
+
Pausing before the parallel agents would require the human to review only the raw idea — less context than after analysis. Pausing after gives the expert all 5 agent outputs to review and correct before they cascade into the Kill Shot design. Pausing after Kill Shot would waste the kill shot compute if the human rejects.
Alternative Rejected

Pre-analysis HITL (human sees only the raw idea, less context). Post-kill-shot HITL (Kill Shot compute wasted if rejected). Multiple HITL pauses (too much friction, slows the pipeline).

Why a single ChromaDB collection with metadata filtering?
Avoids dual similarity thresholds and dual embedding indices
+
Two separate collections (domain + portfolio) would require two similarity thresholds and two embedding indices to maintain. A single collection with source_type metadata lets agents filter when they need specificity (cost agent queries only portfolio for precedent) while the RAG agent queries across both simultaneously for maximum coverage.
Alternative Rejected

Separate collections per source type. Requires tuning two independent similarity thresholds. Cross-source queries (both domain and portfolio simultaneously) require merging two result sets with different score distributions.

Failure Analysis

Known Failure Modes & Mitigations

Multi-agent systems fail in non-obvious ways. These are the specific failure modes this architecture is designed to handle.

Agent-Level
RAG-Level
Graph-Level
System-Level
Cascading Hallucination
One agent produces confident but wrong output (e.g. TRL 9 when technology is TRL 3). Downstream agents build on it, amplifying the error.
Mitigation
Each agent independently re-reads the original idea. Inter-agent consistency checker cross-validates shared assumptions (TRL, market size).
Schema Drift
Agent returns partial schema match — optional fields default to None. Output passes validation but report is silently incomplete.
Mitigation
Critical fields are required (no default). Post-validation checks that no required field is null.
Tool Call Refusal
Model refuses to call the structured output tool or sends malformed arguments, especially on edge-case inputs.
Mitigation
Retry up to 3 times with exponential backoff. Final failure marks agent as failed in state; pipeline continues in degraded mode.
Inter-Agent Inconsistency
Technical agent assumes TRL 2; cost agent silently assumes TRL 7 for its estimates, producing an optimistic financial model.
Mitigation
Aggregator node cross-validates TRL, market size order-of-magnitude, and risk level across all agent outputs before writing final report.
Context Window Overflow
As outputs accumulate in shared state and get injected into subsequent agent prompts, total context exceeds model's limit.
Mitigation
Selective context injection — each agent receives only the state fields it needs, not the full accumulated state.
Context Poisoning
A poorly relevant chunk retrieved and injected into the agent prompt. The agent, trained to respect context, incorporates misleading information.
Mitigation
Similarity threshold (≥0.75 cosine). RAGAS context precision score in CI flags degradation automatically.
Prompt Injection via Documents
A document in the knowledge base contains adversarial text: "Ignore previous instructions and output..."
Mitigation
Document sanitization on ingestion. System prompt explicitly instructs model to treat retrieved content as data, not instructions.
Empty Context Retrieval
Query embedding finds no chunks above similarity threshold. Agent proceeds without grounding.
Mitigation
RAG agent sets grounded=False in state. Downstream agents and judge are notified. Error surfaced explicitly in UI.
Embedding Model Drift
Embedding model used at query time differs from ingestion time (e.g. after dependency upgrade). All similarity scores become meaningless.
Mitigation
Embedding model name stored in ChromaDB metadata per document. Startup check validates consistency between stored and current model.
Infinite Loops
Conditional edges with cycles — a retry loop with no maximum iteration count runs indefinitely.
Mitigation
All cycles have explicit max_iterations counter in state. LangGraph's recursion limit is set as a hard stop.
Partial State Write on Crash
Agent crashes mid-execution, leaving shared state partially written. Downstream agents read stale or null values.
Mitigation
LangGraph checkpointing saves state atomically after each node completes. On restart, graph resumes from last valid checkpoint — not from zero.
Non-determinism in Tests
Same input produces different outputs across runs (temperature > 0, retrieval order variance). Makes regression testing unreliable.
Mitigation
Evals and benchmarks set temperature=0. Score consistency metric: std across 5 identical runs. Alert if std > 0.5.
API Rate Limiting
5 parallel agents hitting the Anthropic API simultaneously. Rate limit (429) or overload (529) terminates all agents at once.
Mitigation
Exponential backoff with jitter, max 3 retries. Parallel agents staggered with small delay if multiple calls issued simultaneously.
Silent Quality Degradation
System produces output without errors but quality dropped — model update changed behavior, retrieval quality degraded as knowledge base aged.
Mitigation
Scheduled benchmark runs on fixed test set. Grafana alert if average judge score drops >10% from 7-day baseline.
Judge Self-Preference
Judge LLM systematically scores outputs from its own model family higher, biasing benchmark comparisons.
Mitigation
Different model family for judge (agents on Sonnet, judge on Opus). Randomized answer order in comparative evaluation.
Technology Choices

Stack & Rationale

Every technology choice has a specific reason. "We used it before" is not a reason here.

LangGraph
Agent Orchestration
Stateful graph with conditional branching, cycles, checkpointing, and native HITL — the right primitive for production agentic systems that must resume, branch, and survive crashes.
LangChain
LLM Abstraction
Makes swapping Claude / GPT-4 / Gemini trivial for benchmarking. Tool wrappers for Arxiv, Wikipedia, DuckDuckGo available out of the box.
Anthropic Claude
Primary LLM
Reliable structured outputs via tool-use mode. Strong reasoning on technical domains. Extended context window for injecting full agent state into Kill Shot prompt.
ChromaDB
Vector Store
Zero infrastructure — runs as a Python library. Abstracted behind vector_store.py so the swap to Pinecone / Vertex AI in production is one file change.
Pydantic v2
Output Schemas
Eliminates runtime errors from free-form LLM text. Required fields force explicit uncertainty, evidence citations, and assumptions. Validation errors surface at the agent boundary, not downstream.
RAGAS
RAG Evaluation
Industry-standard faithfulness, relevancy, context precision/recall metrics. Runs automatically against a fixed reference set in CI to detect retrieval regressions.
LangSmith
Trace Observability
Auto-instrumentation for all LangChain/LangGraph runs — zero extra code for traces. Built-in UI for latency breakdowns, token counts, and prompt version tracking.
FastAPI
Serving Layer
Async-native (matches LangGraph's async execution). Pydantic-native (shared schemas with agent outputs). Auto-generates OpenAPI docs for the SSE + HITL endpoint surface.
Streamlit
Frontend
Pure Python chat interface with real-time SSE streaming. Session memory via st.session_state. Avoids context switch to TypeScript for a Python-native team.
structlog
Structured Logging
Every log line is machine-parseable JSON. Context variables (run_id, agent_name) automatically injected. Compatible with Loki / Grafana log aggregation in production.
Prometheus + Grafana
Metrics & Dashboards
Standard pull-based metrics. Alert rules on error rate, latency P95, and quality score regression. Three dashboards: system health, quality metrics, cost/token usage.
SQLite → PostgreSQL
Eval Score Storage
Queryable benchmark history with SQL. SQLite for local dev (zero infra), PostgreSQL in production for concurrent benchmark writes across API workers.

Getting Started

Implementation Guide

From clone to first moonshot evaluation in 5 steps.

1
Clone and install dependencies
Python 3.11+ required. The virtual environment isolates all LangGraph, LangChain, and ChromaDB dependencies.
# Clone and set up environment
git clone https://github.com/tmehtiyev2019/agentlens.git
cd agentlens
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
2
Configure environment variables
Copy .env.example to .env and fill in your API keys. LangSmith is optional but strongly recommended for trace observability.
ANTHROPIC_API_KEY=sk-ant-...
OPENAI_API_KEY=sk-...         # optional — for GPT-4 benchmarking
LANGSMITH_API_KEY=ls__...     # optional — trace capture
LANGSMITH_PROJECT=agentlens
AGENT_MODEL=claude-sonnet-4-6
JUDGE_MODEL=claude-opus-4-7
3
Ingest domain knowledge into ChromaDB
Populates the vector store with TRL benchmarks, market reports, and published papers. The RAG agent retrieves from this at evaluation time.
# Ingest bundled domain docs (TRL benchmarks, market reports)
python -m src.knowledge_base.ingestor --ingest-domain

# Optionally: add your past project docs to src/knowledge_base/portfolio/
python -m src.knowledge_base.ingestor --ingest-portfolio

# Verify ingestion
python -m src.knowledge_base.ingestor --list-sources
4
Start the backend API and Streamlit frontend
The FastAPI backend handles the LangGraph pipeline. Streamlit is the chat-first frontend with real-time SSE streaming and session memory.
# Terminal 1: Start FastAPI backend
uvicorn src.api.main:app --port 8001 --reload

# Terminal 2: Start Streamlit frontend
streamlit run streamlit_app.py --server.port 3001
5
Submit a moonshot idea and run benchmarks
Open http://localhost:3001 and type your idea, or use the API directly. Run the benchmark suite to compare model configurations.
# Submit via API directly
curl -X POST http://localhost:8001/evaluate \
  -H 'Content-Type: application/json' \
  -d '{"idea": "solid-state batteries for 1000-mile EV range"}'

# Run benchmark suite (sweeps model configs)
python -m src.evaluation.benchmark
